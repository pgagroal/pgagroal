# High Availability

[**pgagroal**][pgagroal] sits in front of [PostgreSQL][postgresql] streaming replication as a connection pool layer.
High availability at this layer requires aligning the replication topology with how [HAProxy][haproxy] routes client traffic.

The setup described in this chapter achieves HA through three cooperating components:

* **[PostgreSQL][postgresql] streaming replication** -- keeps replicas in sync with the primary
* **[**pgagroal**][pgagroal]** -- one instance per server, pooling connections to that server only
* **[HAProxy][haproxy]** -- two endpoints exposed to clients:
  * A **write endpoint** that routes to the primary pgagroal instance
  * A **read endpoint** that load-balances across replica pgagroal instances, and reroutes automatically when a replica goes down

Clients connect only to [HAProxy][haproxy] and are never aware of [**pgagroal**][pgagroal] or [PostgreSQL][postgresql] directly.
When a replica fails, [HAProxy][haproxy] stops routing reads there within seconds -- no client changes needed.
When the primary fails, writes are unavailable until a replica is promoted; see the [Failover](16-failover.md) chapter.

This chapter covers two replication topologies -- Linear and Star -- and the [HAProxy][haproxy] routing strategies that apply to each.
[**pgagroal**][pgagroal] does not manage replication itself; it routes connections to the servers you configure.

> **Note:** All host addresses, ports, usernames, and tuning values used in this chapter are examples only.
> You are free to use any values that fit your environment.

## Prerequisites

* [PostgreSQL][postgresql] streaming replication already configured and running
* [HAProxy][haproxy] installed on a dedicated host
* The host/port layout used throughout this chapter:

| Component          | Role        | Host        | Port |
|--------------------|-------------|-------------|------|
| PostgreSQL         | Primary     | `10.0.0.11` | 5432 |
| PostgreSQL         | Replica 1   | `10.0.0.12` | 5432 |
| PostgreSQL         | Replica 2   | `10.0.0.13` | 5432 |
| pgagroal (write)   | Primary     | `10.0.0.11` | 2345 |
| pgagroal (read 1)  | Replica 1   | `10.0.0.12` | 2345 |
| pgagroal (read 2)  | Replica 2   | `10.0.0.13` | 2345 |
| HAProxy (rw)       | Read-write  | `10.0.0.21` | 5432 |
| HAProxy (ro)       | Read-only   | `10.0.0.22` | 5433 |

## Health Check

[HAProxy][haproxy] probes each [**pgagroal**][pgagroal] instance to decide whether to route traffic to it.
[**pgagroal**][pgagroal] probes the [PostgreSQL][postgresql] backend directly.
[HAProxy][haproxy] does not contact [PostgreSQL][postgresql] directly.

A single [PostgreSQL][postgresql] role -- `pgagroal_health` -- is used for both layers. Health probes connect to a dedicated database created specifically for this purpose, as described in the [Health Check](19-health_check.md) chapter.
Before continuing, complete that setup -- create the `pgagroal_health` role and its database, register the role with each [**pgagroal**][pgagroal] instance, and verify it can connect. The configurations in this chapter assume that setup is already in place.

> **Note:** For the [HAProxy][haproxy] probe to succeed, the `pgagroal_health` role must be registered in **PostgreSQL** (`CREATE ROLE`), **pgagroal_users.conf**, and authorized in both **pg_hba.conf** and **pgagroal_hba.conf**.

### pgagroal Configuration

Enable health checks in `pgagroal.conf` on every pgagroal instance:

```ini
[pgagroal]
health_check = on
health_check_user = pgagroal_health
```

### HAProxy Configuration

Add the following to every `backend` block in your HAProxy configuration files:

```
option pgsql-check user pgagroal_health
```

### Tuning

The `check` parameters on each `server` line control probe behavior:

```
server pgagroal-primary 10.0.0.11:2345 check inter 3s rise 2 fall 3
```

| Parameter | Meaning                                        |
|-----------|------------------------------------------------|
| `inter`   | Probe interval (e.g. `3s` = every 3 seconds)  |
| `rise`    | Consecutive successes before marking UP        |
| `fall`    | Consecutive failures before marking DOWN       |

The values above (`inter 3s rise 2 fall 3`) mean: probe every 3 seconds, mark UP after 2 successes, mark DOWN after 3 failures.

## Replication Topologies

### Linear (Cascading)

In a linear topology each replica replicates from the node directly above it, forming a chain:

```
Primary -> Replica 1 -> Replica 2
```

Replica 2 does not receive data from the primary directly; it receives it through Replica 1.
This is known as cascading replication.

**Configuring a replica to replicate from another replica**

**Step 1 -- Create a replication slot on Replica 1** (the upstream node for Replica 2):

```sql
SELECT pg_create_physical_replication_slot('replica2_slot');
```

> **Note:** Replication slots prevent the upstream node from discarding WAL before the replica has consumed it.
> A slot for a down replica causes WAL to accumulate indefinitely -- monitor slot lag in production.

**Step 2 -- Configure Replica 2** by setting `primary_conninfo` and `primary_slot_name` in `postgresql.conf`:

```ini
primary_conninfo = 'host=10.0.0.12 port=5432 user=replicator password=secret'
primary_slot_name = 'replica2_slot'
```

Restart Replica 2 for the change to take effect. Replica 1's configuration remains unchanged -- it still replicates from the primary.

This cascading setup is referenced in the Two Level star section below.

---

#### Backup

Replica 2 acts as a spare. Replica 1 handles all read traffic under normal conditions.
If Replica 1 goes down, [HAProxy][haproxy] automatically promotes Replica 2 to active and routes reads there.

**Pros**

* Minimal read downtime -- [HAProxy][haproxy] promotes Replica 2 within `fall x inter` seconds of detecting Replica 1 failure
* Simple to configure

**Cons**

* Replica 2 is idle under normal conditions
* After Replica 1 fails, Replica 2 is no longer receiving updates from the primary through Replica 1 -- data becomes stale
* DBAs must restore Replica 1 quickly to avoid serving outdated reads

**PostgreSQL configuration**

No changes to the replication setup described above. Replica 2 continues to replicate from Replica 1.

![Linear topology -- Backup (PostgreSQL level)](images/ha-linear-backup-pg.png){width=50%}

**HAProxy configuration**

Mark Replica 2 as `backup` in `haproxy-ro.cfg`:

`haproxy-ro.cfg`:
```
backend be_pgagroal_replicas
    balance leastconn
    option pgsql-check user pgagroal_health
    server pgagroal-replica1 10.0.0.12:2345 check inter 3s rise 2 fall 3
    server pgagroal-replica2 10.0.0.13:2345 check inter 3s rise 2 fall 3 backup
```

![Linear topology -- Backup (HAProxy + pgagroal)](images/ha-linear-backup-haproxy.png){width=90%}

---

#### Partner

Both replicas share read traffic equally. Because Replica 2 depends on Replica 1 for data,
their health is tied together using `track`. If Replica 1 goes down, [HAProxy][haproxy] immediately
takes Replica 2 offline as well.

**Pros**

* Clients never read stale data -- Replica 2 is taken offline the moment its upstream fails
* Both replicas are active under normal conditions

**Cons**

* Read downtime when Replica 1 fails -- clients receive connection errors until DBAs restore it
* Replica 2 capacity is lost even though Replica 2 itself is still running

**PostgreSQL configuration**

Same cascading setup as above. No additional changes required.

![Linear topology -- Partner (PostgreSQL level)](images/ha-linear-partner-pg.png){width=50%}

**HAProxy configuration**

Use `track` on Replica 2 in `haproxy-ro.cfg`:

`haproxy-ro.cfg`:
```
backend be_pgagroal_replicas
    balance leastconn
    option pgsql-check user pgagroal_health
    server pgagroal-replica1 10.0.0.12:2345 check inter 3s rise 2 fall 3
    server pgagroal-replica2 10.0.0.13:2345 track be_pgagroal_replicas/pgagroal-replica1
```

![Linear topology -- Partner (HAProxy + pgagroal)](images/ha-linear-partner-haproxy.png){width=90%}

### Star

In a star topology every replica replicates directly from the primary:

```
        Primary
       /       \
Replica 1   Replica 2
```

Each replica is independent. Losing one replica does not affect the other.

---

#### Single Level

All replicas replicate directly from the primary and share the read load equally.
This is the simplest topology to operate and reason about.

**Pros**

* No read downtime -- losing one replica leaves the other fully operational
* Data on surviving replicas is always fresh
* Simple configuration

**Cons**

* Losing the primary takes down writes; replicas continue serving reads from the last committed data
* No protection if both replicas fail simultaneously

**PostgreSQL configuration**

On Replica 1 and Replica 2, `primary_conninfo` points at the primary:

```ini
primary_conninfo = 'host=10.0.0.11 port=5432 user=replicator password=secret'
```

![Star topology -- Single Level (PostgreSQL level)](images/ha-star-single-pg.png){width=65%}

**HAProxy read-write configuration** (`haproxy-rw.cfg`):

```
global
    maxconn     4000
    daemon

defaults
    mode        tcp
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend fe_pg_write
    bind *:5432
    default_backend be_pgagroal_primary

backend be_pgagroal_primary
    option pgsql-check user pgagroal_health
    server pgagroal-primary 10.0.0.11:2345 check inter 3s rise 2 fall 3

listen stats
    mode http
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 5s
```

**HAProxy read-only configuration** (`haproxy-ro.cfg`):

[HAProxy][haproxy] supports several load balancing algorithms for the read backend. The choice of the right algorithm depends on your specific use case:

| Algorithm    | Behavior                                              | Use when                                  |
|--------------|-------------------------------------------------------|-------------------------------------------|
| `roundrobin` | Cycles through replicas in turn                       | Uniform read workloads on replicas with identical hardware capacity |
| `leastconn`  | Routes to the replica with the fewest active connections | Mixed workloads with long-running queries or transactions |
| `source`     | Same client IP always hits the same replica           | Session consistency is required (e.g. read-your-own-writes) |


```
global
    maxconn     4000
    daemon

defaults
    mode        tcp
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend fe_pg_read
    bind *:5433
    default_backend be_pgagroal_replicas

backend be_pgagroal_replicas
    balance leastconn
    option pgsql-check user pgagroal_health
    server pgagroal-replica1 10.0.0.12:2345 check inter 3s rise 2 fall 3
    server pgagroal-replica2 10.0.0.13:2345 check inter 3s rise 2 fall 3

listen stats
    mode http
    bind *:8405
    stats enable
    stats uri /stats
    stats refresh 5s
```

![Star topology -- Single Level (HAProxy + pgagroal)](images/ha-star-single-haproxy.png){width=100%}

---

#### Two Level

The two level topology adds a second tier of replicas to protect against losing an entire branch.
Each first-level replica has two second-level replicas replicating from it:

```
              Primary
             /       \
       Replica 1   Replica 2
       /     \     /      \
 Replica1-1 Replica1-2  Replica2-1  Replica2-2
```

> **Note:** This topology requires four additional hosts (`10.0.0.14`--`10.0.0.17`) for the second-level replicas, beyond the prerequisites table above.

Losing Replica 1 affects only Replica 1-1 and Replica 1-2. The Replica 2 branch continues serving reads uninterrupted.
This is the only way to achieve full HA on the replica tier itself in a star topology.

Second-level replicas should be configured as **Partner** -- their health is tied to their upstream first-level replica.

**Pros**

* Full HA on both the primary tier and the replica tier
* Losing one entire branch does not affect reads on the other branch
* Second-level replicas can serve reads, spreading load further

**Cons**

* Higher operational complexity -- more instances to monitor and maintain
* Second-level replicas lag further behind the primary (two hops)

**PostgreSQL configuration**

First-level replicas (Replica 1, Replica 2) replicate from the primary as in Single Level above.

Second-level replicas (Replica 1-1, Replica 1-2) replicate from Replica 1; Replica 2-1 and Replica 2-2 replicate from Replica 2.
Configure each as described in the [cascading replication section above](#linear-cascading):

```ini
# Replica 1-1 and Replica 1-2: replicate from Replica 1
primary_conninfo = 'host=10.0.0.12 port=5432 user=replicator password=secret'

# Replica 2-1 and Replica 2-2: replicate from Replica 2
primary_conninfo = 'host=10.0.0.13 port=5432 user=replicator password=secret'
```

![Star topology -- Two Level (PostgreSQL replication tree)](images/ha-star-two-level-pg.png){width=100%}

**HAProxy configuration**

The [HAProxy][haproxy] configuration is the same structure as Single Level, extended to include
the second-level replicas. Each second-level replica tracks its upstream first-level replica using `track`:

`haproxy-ro.cfg`:
```
backend be_pgagroal_replicas
    balance leastconn
    option pgsql-check user pgagroal_health
    server pgagroal-replica1  10.0.0.12:2345 check inter 3s rise 2 fall 3
    server pgagroal-replica2  10.0.0.13:2345 check inter 3s rise 2 fall 3
    server pgagroal-replica1-1 10.0.0.14:2345 track be_pgagroal_replicas/pgagroal-replica1
    server pgagroal-replica1-2 10.0.0.15:2345 track be_pgagroal_replicas/pgagroal-replica1
    server pgagroal-replica2-1 10.0.0.16:2345 track be_pgagroal_replicas/pgagroal-replica2
    server pgagroal-replica2-2 10.0.0.17:2345 track be_pgagroal_replicas/pgagroal-replica2
```

When Replica 1 goes down, [HAProxy][haproxy] immediately removes Replica 1-1 and Replica 1-2 from rotation.
The Replica 2 branch (and Replica 2-1, Replica 2-2) continues serving reads normally.

![Star topology -- Two Level (HAProxy + pgagroal)](images/ha-star-two-level-haproxy.png){width=100%}

## Best Practices

### Choosing a Topology

* If you need a **simple setup with independent replicas**, choose the *Star Single Level* topology.
  Losing one replica does not affect the other, configuration is minimal, and it is the easiest topology to operate.

* If you need **full HA on both the primary and replica branches**, choose the *Star Two Level* topology.
  Each branch survives independently. Configure second-level replicas with `track` so they are taken offline when their upstream replica fails, preventing stale-data reads.

* If you are using **cascading replication and prefer read availability over consistency**, choose the *Linear Backup* topology.
  Replica 2 covers reads when Replica 1 fails, but data may be stale until Replica 1 is restored.
  DBAs must act quickly -- the longer Replica 1 is down, the more stale Replica 2 becomes.

* If you are using **cascading replication and prefer consistency over availability**, choose the *Linear Partner* topology.
  Reads go down when Replica 1 fails, but clients never see stale data. Use this only when brief read downtime is acceptable.

* If you are **not sure where to start**, choose the *Star Single Level* topology. It is the simplest to reason about and covers most use cases.

### General Tips

* Prefer Star over Linear where possible -- the failure blast radius is smaller and recovery is simpler.
* Enable `health_check` on all [**pgagroal**][pgagroal] instances, not just those behind [HAProxy][haproxy].
* Monitor replica lag continuously. In Linear Backup, a lagging Replica 2 means a larger stale-data window if Replica 1 fails.
* In Linear topologies, when Replica 1 fails, Replica 2 stops receiving new data. Restore Replica 1 before the lag becomes operationally significant.
* Test your HAProxy failover regularly -- send traffic, kill a replica, confirm [HAProxy][haproxy] reroutes within `fall x inter` seconds.
